GitOps Principles

Nguồn: OpenGitOps.dev | CNCF GitOps Working Group

1. OpenGitOps là gì?

Không phải tool — mà là bộ nguyên tắc định nghĩa "thế nào mới là GitOps".

OpenGitOps = Luật chơi
Argo CD    = Công cụ tuân theo luật đó
Flux CD    = Công cụ tuân theo luật đó

2. Trước và sau GitOps

❌ Trước (Push model):
Dev → CI/CD Pipeline → kubectl apply → Kubernetes
                            ↑
                     pipeline cần giữ credential cluster

✅ Sau (GitOps):
Dev → git push → Git Repository
                       ↑
              Argo CD / Flux tự kéo về → Kubernetes

Điểm khác biệt: Không ai deploy trực tiếp vào cluster — chỉ sửa Git.

3. 4 Nguyên tắc cốt lõi

Principle 1: Declarative — Mô tả trạng thái, không ra lệnh

# ✅ Declarative (GitOps) — Mô tả "muốn gì"
spec:
  replicas: 3

# ❌ Imperative (không phải GitOps) — Ra lệnh "làm gì"
kubectl create pod

Principle 2: Versioned & Immutable — Lưu trong Git, có lịch sử

Git log:
  commit A — replicas: 2  (Nguyen, 2025-06-01)
  commit B — replicas: 5  (Tran,   2025-06-08)

Mọi thay đổi phải đi qua Git commit — không ngoại lệ.

Principle 3: Pulled Automatically — Cluster tự kéo, không phải Pipeline đẩy

Tiêu chí Push (CI/CD cũ) Pull (GitOps)
Credential CI/CD giữ key của cluster Cluster tự giữ key của Git
Bảo mật Bị hack CI/CD là mất cả cluster Cluster chỉ cần chiều kéo ra ngoài

Principle 4: Continuously Reconciled — Tự sửa drift liên tục

Nếu ai đó kubectl scale bằng tay trên cluster, Agent sẽ phát hiện sự sai lệch (Drift) và tự động kéo về trạng thái chuẩn trên Git.

Argo CD

GitOps Continuous Delivery tool for Kubernetes — Git là nguồn sự thật duy nhất.

1. Kiến trúc

Web UI / CLI / CI
       ↓
  API Server  ──→  Repo Server (clone Git, render Helm/Kustomize)
       ↓
App Controller (so sánh Git vs Cluster → OutOfSync? → Sync)
       ↓
    Redis (cache)

2. Thuật ngữ cốt lõi

Thuật ngữ Ý nghĩa
Application CRD: "deploy cái gì, từ đâu, vào đâu"
Sync Đưa cluster về đúng trạng thái Git
Drift Cluster bị ai đó sửa tay → lệch với Git

3. Application — YAML mẫu

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/myrepo.git
    targetRevision: HEAD   
    path: k8s/my-app       
  destination:
    server: https://kubernetes.default.svc
    namespace: my-namespace
  syncPolicy:
    automated:
      prune: true       # xóa resource khi Git xóa
      selfHeal: true    # tự phục hồi khi bị sửa tay

4. Các Tính năng Nâng cao

  • AppProject: Phân quyền đa team, giới hạn repo và cluster.
  • ApplicationSet: Tạo hàng loạt Application tự động (rất tốt cho multi-cluster).
  • App of Apps Pattern: Một Root App quản lý nhiều Child Apps để bootstrap toàn bộ cluster.
  • Sync Waves & Hooks: Kiểm soát thứ tự deploy (ví dụ chạy DB Migration trước khi deploy App).

Flux CD

Kubernetes-native GitOps tool | Cấu trúc phân tán với nhiều Controller

1. Kiến trúc — Các Controller

Controller Nhiệm vụ CRD tương ứng
Source Controller Clone Git/Helm repo, tạo artifact GitRepository, HelmRepository
Kustomize Controller Đọc YAML → apply vào cluster Kustomization
Helm Controller Deploy/upgrade/rollback Helm chart HelmRelease

2. Ba CRD quan trọng nhất

GitRepository

apiVersion: source.toolkit.fluxcd.io/v1
kind: GitRepository
metadata:
  name: my-app
spec:
  interval: 1m
  url: https://github.com/myorg/myrepo
  ref:
    branch: main

Kustomization

apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
spec:
  interval: 10m
  path: ./k8s/apps/my-app
  sourceRef:
    kind: GitRepository
    name: my-app
  prune: true

3. Lệnh Bootstrap — Bước quan trọng nhất

flux bootstrap github \
  --owner=myorg \
  --repository=fleet-infra \
  --branch=main \
  --path=clusters/my-cluster \
  --personal

Sau lệnh này, Flux tự động cài tất cả controllers lên cluster và thiết lập cầu nối Git ↔ Cluster.

4. So sánh Flux vs Argo CD

Tiêu chí Flux Argo CD
Kiến trúc Nhiều controller nhỏ gọn 1 ứng dụng lớn tập trung
Giao diện (UI) Không có sẵn (dùng WeaveGitOps) Web UI cực kỳ mạnh và trực quan
Image auto-update Có sẵn (Built-in) Cần cài thêm Image Updater

GitHub Actions

Robot tự động làm việc thay cho bạn sau mỗi sự kiện trên GitHub (CI/CD).

1. Bức tranh tổng thể

Workflow (.github/workflows/ci.yml)
    │
    ├── Job: build        ← chạy trên máy riêng (Runner)
    │     ├── Step: checkout code
    │     ├── Step: npm install
    │     └── Step: npm build
    │
    └── Job: deploy
          ├── Step: login Docker
          └── Step: push image

2. Cấu trúc file Workflow

name: CI Pipeline

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4   # Action: clone code về
      
      - name: Install dependencies
        run: npm install            # Script thường
        
      - name: Run tests
        run: npm test

3. Tích hợp trong DevOps hiện đại

Developer push code → GitHub
    ↓
GitHub Actions (CI)
    ├── Build & Test
    ├── Build Docker Image → Push Docker Hub
    └── Update K8s manifest (Sửa file deployment.yaml)
         ↓
      Argo CD phát hiện thay đổi trên Git
         ↓
      Kéo code mới và Deploy vào Kubernetes

4. Các tính năng hữu ích

  • Secrets: Lưu trữ key/token an toàn ${{ secrets.AWS_KEY }}.
  • Cache: Tăng tốc quá trình build bằng cách lưu lại dependencies.
  • Matrix: Chạy test song song trên nhiều môi trường (nhiều version Node/Python).
  • Self-hosted Runner: Chạy workflow trên máy chủ riêng của công ty thay vì máy của GitHub.
